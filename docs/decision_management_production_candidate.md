# Decision Management Production Candidate

Decision Management is the second production-candidate Working Context. Evidence Review answers, "What do we know?" Decision Management answers, "What should we do?"

The goal is not to create another module. The goal is to validate whether the Working Context Framework can produce another room using the same grammar: orientation, working set, focal object, detail, progressive depth, bounded actions, and adjacent task transitions.

## Decision Room

The Decision Room composes existing state from Semantic Intelligence, Decision Valuation, Decision Workflow, and Evidence Review. It does not own canonical decision data. Durable decision contexts, alternatives, recommendations, valuation records, workflow records, evidence references, and approvals remain owned by their existing services.

The room exposes:

- Decision frame
- Current Decision
- Alternatives
- Tradeoffs
- Economics
- Governance
- Decision detail
- Contextual mentor
- Evidence and economics depth
- Diagnostics and architecture depth

## Decision Studio

Decision Studio follows the same spatial grammar as Evidence Review.

- Left rail: alternatives as first-class decision objects.
- Center canvas: Current Decision, teaching strip, tradeoffs, economics, and governance.
- Right inspector: provenance, workflow state, and contextual mentor.
- Bottom depth: action table, recommendation history, evidence references, valuation records, workflow summary, and cross-context comparison.

The focal object is Current Decision. This is intentionally parallel to Evidence Review's Current Answer.

## Cross-Context Validation

The two rooms intentionally share structure while answering different questions.

| Dimension | Evidence Review | Decision Management |
| --- | --- | --- |
| Primary question | What do we know? | What should we do? |
| Focal object | Current Answer | Current Decision |
| Working material | Evidence, contradictions, synthesis, gaps | Alternatives, tradeoffs, valuation, recommendation, governance |
| Output | Evidence sufficiency and supported next action | Governed recommendation state and decision readiness |
| Governance role | Prevent unsupported conclusions | Keep authority, uncertainty, approval, and implementation visible |

This is the first real test of whether Working Context is a framework rather than an Evidence Review-specific page pattern.

## Founder Review

Review should focus on:

- Does the room naturally follow Evidence Review?
- Does it answer "What should we do?" before exposing tables?
- Do alternatives feel like real decision objects?
- Are benefits, costs, risks, unknowns, opportunity cost, and optionality legible?
- Does governance feel supportive rather than intrusive?
- Does the mentor clarify rather than drive the work?
- Does the user know when to return to Evidence Review?

## Campaigns

Candidate improvement campaigns:

1. Decision object clarity: make the Current Decision understandable in under ten seconds.
2. Alternatives hierarchy: make preferred, baseline, deferred, and evidence-seeking options visually distinct.
3. Tradeoff comprehension: separate benefits, costs, risks, unknowns, opportunity cost, and optionality without table overload.
4. Governance support: make review and approval status feel like decision readiness rather than bureaucracy.
5. Evidence handoff: make return to Evidence Review obvious when a recommendation is not justified.
6. Outcome learning: make implemented decisions visibly lead to outcome review.

## Framework Observations

The Working Context Framework generalized without requiring a new architecture. The necessary refinement was cross-context comparison: a lightweight contract that explains how two rooms can share structure while preserving different focal objects and questions.

The third Working Context should be built only after founder review confirms that Decision Management feels like another room in the same house.
