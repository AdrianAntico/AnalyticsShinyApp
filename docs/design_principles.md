# Analytics Workstation Design Principles

## Evidence Before Conclusions

The system should collect, preserve, and expose evidence before presenting recommendations as decision-relevant.

## Govern Investigations, Not Just Outputs

The important product object is the investigation path: objective, uncertainty, explanations, evidence, belief revision, recommendation, verification, and readiness.

## Deterministic Knowledge First

Deterministic facts should be computed deterministically. Probabilistic reasoning should be reserved for ambiguity, synthesis, judgment, and uncertain prioritization.

## Make Uncertainty Visible

Uncertainty, limitations, rejected explanations, contradictory evidence, and assumptions should remain visible instead of being hidden behind a polished answer.

## Recommendations May Evolve

A recommendation that changes because evidence changed is a strength, not a failure. The system should show how and why it changed.

## Every Claim Should Be Traceable

Claims should point back to evidence, diagnostics, methodology, assumptions, and limitations.

## Challenge Conclusions Before Trust

The system should search for reasons its own recommendation could be wrong before asking the user to act.

## Use AI Inside Contracts

AI should operate through bounded provider, action, evidence, and mutation contracts. It should not receive arbitrary authority merely because it can produce fluent language.

## Remove Unnecessary Limits

The workstation should feel capable and expansive while respecting real constraints: privacy, local-first operation, governance, reproducibility, and user control.

## Reward Curiosity

The product should feel crafted. Visual atmosphere, elegant interactions, and polished evidence surfaces are part of making serious analytical work enjoyable.
