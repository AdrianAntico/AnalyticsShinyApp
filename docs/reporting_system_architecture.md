# Reporting System Architecture

## Status

This document defines the target architecture for Analytics Workstation reporting.

It is a contract and migration guide, not a broad runtime implementation plan. The current Layout and Export pages remain valid application surfaces. They should migrate incrementally toward the contracts below rather than being replaced in one disruptive pass.

## Executive Summary

Reports are first-class human analytical interfaces. They are not artifacts, exports, screenshots, RMarkdown files, or layout grids.

The corrected architecture is:

```text
Analysis Service
  -> CanonicalAnalysisResult
       -> Artifact Service
            ArtifactBundle
            machine-facing evidence
       -> Report Service
            ReportContract
            human-facing communication
              -> ReportRenderer + PresentationProfile
                    interactive app
                    HTML
                    PDF
                    DOCX
                    PowerPoint
                    print
```

Artifacts and reports are sibling outputs derived from the same canonical analysis result. Reports may cite artifacts, and artifacts may provide backing evidence, but reports are not generated by scraping artifacts. The canonical analysis result remains the source of analytical truth.

## Architecture Review Findings

The existing repository already contains strong foundations: artifact bundles, report plans, render targets, information encoding, table artifact policies, quality metadata, and app delivery surfaces.

The missing layer is a durable human-report contract that can survive package conversion, renderer changes, and future Python/API consumers.

| Question | Finding | Architectural Response |
| --- | --- | --- |
| Are reports independent of exports? | Partially. Current report plans mostly curate artifacts for layout/export. | Introduce `ReportContract` as the semantic human-facing report object. |
| Is presentation ownership clear? | Not yet. Visual choices are mixed with page/layout behavior. | Separate `ReportContract` from `PresentationProfile`. |
| Can reports be interactive and static? | Yes in principle, but not contractually guaranteed. | Require renderer degradation rules for every interactive component. |
| Can a future package expose this cleanly? | Not without a stable contract. | Use JSON-serializable S3/list contracts with explicit validation and versioning. |
| Are tables treated as report-native components? | Tables exist as artifacts, previews, and UI outputs. | Add report table components with fit-to-meaning presentation rules. |
| Can guided/studio modes share work? | Not yet formal. | Make Automatic, Guided, and Studio modes produce or edit the same `ReportContract`. |

## Core Principle

Separate truth, communication, presentation, and delivery.

```text
Truth
  CanonicalAnalysisResult

Communication
  ReportContract

Presentation
  PresentationProfile

Delivery
  ReportRenderer
```

A renderer may decide how something looks in HTML or PDF. It must not decide what the report means.

## Formal Object Definitions

### CanonicalAnalysisResult

`CanonicalAnalysisResult` is the deterministic output of an analysis run before downstream representation.

It owns analysis identity, module identity, run identity, input references, validated configuration, computed data products, model objects or model references, metrics, diagnostics, tables, plot specifications or plot objects, findings, recommendations, warnings, lineage, and reproducibility metadata.

It does not own human layout, export styling, or artifact sidecar policy.

### ArtifactBundle

`ArtifactBundle` is the machine-facing evidence representation derived from a canonical analysis result.

It owns artifact identity, artifact type, captions, quality metadata, producer semantics, screenshots, table sidecars, JSON sidecars, diagnostics, recommendations, collector metadata, and render-target metadata for machine or evidence workflows.

It does not own human report structure.

### ReportContract

`ReportContract` is the versioned semantic contract for a human-facing report.

It owns report identity, contract version, audience, purpose, communication mode, narrative structure, sections, components, findings, recommendations, evidence links, interaction descriptors, static fallback requirements, provenance, and validation status.

It is not HTML, Shiny UI, RMarkdown, PDF, DOCX, or a plot grid.

### ReportComponent

`ReportComponent` is a semantic communication unit inside a report section.

Examples include title blocks, orientations, executive summaries, findings, recommendations, metric summaries, visualizations, tables, diagnostic disclosures, methodology notes, evidence citations, and technical appendices.

A component may reference artifact IDs, but it is not identical to an artifact.

### ReportRenderer

`ReportRenderer` turns a `ReportContract` and `PresentationProfile` into a delivery target.

It owns delivery-specific layout, responsive behavior, static fallback rendering, export mechanics, accessibility adaptation, and target-specific limitations.

It must not recompute analysis or reinterpret findings.

### PresentationProfile

`PresentationProfile` is the report presentation policy.

It owns typography, spacing density, theme, table density, chart framing, executive/technical depth, page or slide constraints, interaction affordance style, and export polish.

Presentation is a distinct layer. In the near term it can be implemented inside the Report Service as a profile subsystem. It should not become a separate service until multiple report families and renderers justify the extra boundary.

## Versioning

`ReportContract` must be versioned from the first implementation.

Recommended initial fields:

- `contract_version`: semantic contract version, for example `0.1.0`
- `schema_version`: validator/schema version, for example `report_contract_v0_1`
- `producer_version`: package or app version that produced it
- `renderer_min_version`: optional renderer compatibility floor

Compatibility rules:

- Minor versions may add optional fields.
- Patch versions may clarify validation but must not break valid existing contracts.
- Major versions may break structure only with explicit migration functions.
- Renderers must reject unsupported major versions with clear messages.
- Validators should return structured errors and warnings, not unhandled exceptions.

## Phase 1 Runtime Foundation

The Phase 1 runtime lives in `R/report_contract.R`.

It establishes the semantic reporting language only:

- `create_canonical_analysis_result()`
- `create_report_contract()`
- `create_report_component()`
- report component constructors
- `create_presentation_profile()`
- `validate_report()`
- `validate_components()`
- `validate_contract_version()`
- `validate_capabilities()`
- `serialize_report()`
- `deserialize_report()`
- `report_summary()`
- `qa_report_contract_runtime()`

It intentionally contains no renderer, UI, HTML, PDF, DOCX, PowerPoint, or Shiny-specific behavior.

The platform invariant is:

```text
Every complete Analytics Service shall produce:
  CanonicalAnalysisResult
  ArtifactBundle
  ReportContract
```

The helper `validate_analytics_service_outputs()` checks that invariant for future service integrations.

Phase 2 adds the first concrete producer in `R/report_regression_model_insights.R`.

The Regression Model Insights adapter proves the contract by translating existing module output into semantic report components. It does not run analysis, render UI, export files, or call AutoQuant-native report rendering.

The adapter exposed one contract refinement: findings need to be first-class semantic objects, not loose lists. The runtime now includes `create_report_finding()`, `validate_report_finding()`, and `validate_report_findings()`.

## Component Model

Report components should be small, composable, and semantic. They should describe what they communicate before how they render.

| Component | Purpose | Required Fields | Static Fallback |
| --- | --- | --- | --- |
| `report_title` | Identify the report and context. | title, subtitle, source, timestamp | Plain title block |
| `orientation` | Explain what the reader is about to see. | question, scope, audience | Narrative block |
| `executive_summary` | Summarize conclusions and next action. | summary, confidence, next_action | Narrative block |
| `finding` | State an analytical conclusion. | statement, support, caveat, confidence | Finding card |
| `recommendation` | State a proposed action. | action, rationale, risk, evidence | Recommendation block |
| `metric_summary` | Present key numbers. | metrics, labels, units | Table or metric list |
| `visualization` | Show a plot or visual explanation. | plot_ref or spec, caption, interpretation | Static image or text summary |
| `table` | Present structured values. | table_ref or data, table_contract | Static table |
| `diagnostic` | Show caveats, warnings, and checks. | status, messages, severity | Disclosure block |
| `methodology` | Explain method and assumptions. | method, assumptions, limitations | Narrative block |
| `evidence_link` | Link to supporting artifacts. | artifact_id, role, relationship | Citation list |
| `technical_appendix` | Preserve reproducibility detail. | content, references | Appendix section |

Each component should support `semantic_role`, `importance`, `audience_level`, `evidence_links`, `interaction_descriptor`, `static_fallback`, and `presentation_hints`.

## Table Architecture For Reports

Report tables are not raw data dumps. They are human communication components.

The table contract should include table id, title, purpose, source result id, source artifact id when available, row meaning, column meaning, units, default sort, alternate sort views, priority columns, pinned columns, formatting rules, conditional emphasis, row grouping, summarization policy, max rows for report view, export expansion policy, static fallback policy, and accessibility label.

Presentation rules:

- Fit to meaning, not to available pixels.
- Use compact density for analyst reports and higher whitespace for executive reports.
- Preserve horizontal scroll only when the table is genuinely wide.
- Pin identifiers when horizontal scroll is unavoidable.
- Prefer ranked, grouped, or summarized views over raw wide tables.
- Separate preview tables from full backing data.
- Never let a renderer silently drop columns without recording the degradation.

## Renderer Degradation Hierarchy

Every interactive report component must declare how it degrades.

Recommended hierarchy:

```text
Interactive Workstation
  -> Standalone Interactive HTML
       -> Static HTML
            -> PDF
                 -> DOCX
                      -> PowerPoint
                           -> Plain Text Summary
```

The hierarchy is not a quality ranking. It describes expected capability loss.

| Capability | Workstation | HTML | PDF | DOCX | PPTX |
| --- | --- | --- | --- | --- | --- |
| expand/collapse diagnostics | yes | yes | rendered open/closed | rendered open/closed | summarized |
| sortable table | yes | yes | default sort | default sort | selected rows |
| chart hover | yes | yes | static annotation | static image | static image |
| linked evidence drilldown | yes | link | citation | citation | note |
| responsive layout | yes | yes | page constrained | page constrained | slide constrained |

Renderers must report degradations in metadata.

## Progressive Complexity

The same contract supports three authoring modes.

### Automatic

The system creates a valid report contract from the canonical analysis result using defaults. The user receives a useful report without opening a builder.

### Guided

The system presents a small set of meaningful choices: audience, purpose, depth, emphasis, included sections, and export target. The output is still a validated `ReportContract`.

### Studio

The user edits sections, components, ordering, narrative emphasis, and presentation profile. Studio mode must edit the same contract. It must not create a separate report model.

## Service Boundary

### Analysis Service

Runs analysis and returns `CanonicalAnalysisResult`.

### Artifact Service

Converts canonical results into `ArtifactBundle` objects and collector-ready evidence.

### Report Service

Converts canonical results into `ReportContract` objects, validates them, and stores report intent.

### Renderer Registry

Chooses the renderer for a target and reports target capability.

### Renderer

Renders a contract with a presentation profile.

## Public API Sketch

The package-facing API should remain small.

```r
result <- run_analysis(...)

contract <- create_report_contract(
  result,
  report_type = "regression_model_insights",
  audience = "analyst",
  purpose = "diagnostic_review",
  mode = "automatic"
)

validation <- validate_report_contract(contract)

view <- render_report(
  contract,
  target = "workstation",
  profile = presentation_profile("workstation_dark")
)

export_report(
  contract,
  target = "html",
  path = "report.html",
  profile = presentation_profile("executive")
)
```

Initial implementation should prefer S3 list objects with validators and JSON serialization. R6 is unnecessary until mutable report editing becomes complex enough to justify it.

## Relationship To Current Report Plans

Current report plans are useful migration scaffolding.

They should become one source of section ordering, one source of included artifact references, and one input to `ReportContract`.

They should not remain the primary report abstraction.

## First Vertical Slice Recommendation

The first vertical slice should be Regression Model Insights.

Reasons:

- It already has an app module and module documentation.
- It produces a broad but bounded mix of metrics, effects, diagnostics, tables, plots, warnings, and report plans.
- It is more representative than a simple EDA table.
- It is less broad than full EDA.
- It is less specialized than SHAP.
- It can demonstrate the split between artifacts, report contract, and renderers without forcing a full reporting runtime.

## Migration Plan

### Phase 0: Contract Approval

Agree on the `ReportContract` fields, component set, and first slice.

### Phase 1: Contract Skeleton

Implement constructors, validators, JSON serialization, fixtures, and no-op renderer metadata.

### Phase 2: Regression Model Insights Adapter

Create `create_regression_model_insights_report_contract()` from a canonical result or existing module output.

Implemented as `build_regression_model_insights_report()`.

### Phase 3: Multi-Service Reporting Generalization

Prove that structurally different Analytics Services can produce the same `ReportContract` shape without renderer logic or service-specific reporting models.

Implemented adapters:

- `build_regression_model_insights_report()`
- `build_shap_analysis_report()`
- `build_eda_report()`

The shared `R/report_adapter_helpers.R` layer contains adapter-safe utilities for artifact indexing, evidence links, semantic table descriptors, semantic visualization descriptors, source-supplied findings, optional recommendations, and section finalization.

The generalization pass exposed one real runtime weakness: section title lookup assumed every generated section ID existed in a supplied title map. That was too brittle once SHAP and EDA generated domain-specific section IDs. The shared finalizer now degrades to a generated title instead of failing, and generated artifact sections are namespaced by adapter prefix to avoid cross-service collisions.

No new report component type was required for SHAP or EDA. SHAP importance, dependence, interaction, effect-curve, segment, time, and local explanation evidence fit the existing `visualization`, `table`, `finding`, `diagnostic`, `methodology`, and `evidence_link` components. EDA missingness, distributions, correlations, trends, target analysis, drift diagnostics, and leakage/risk flags fit the same model.

### Phase 4: Interactive Report Browser

Render validated `ReportContract` objects inside Analytics Workstation using a native component-renderer registry.

Implemented files:

- `R/report_browser.R`
- `R/page_report_browser.R`
- `tests/testthat/test-report-browser.R`

The browser consumes only the semantic contract:

```text
ReportContract
  -> sections
  -> components
  -> presentation_profile
  -> capabilities
  -> native workstation browser
```

The renderer does not call Regression, SHAP, or EDA code paths. Those services remain upstream adapter producers. The browser renders by component type through `report_browser_component_registry()`, with public wrappers `render_report()`, `render_section()`, and `render_component()`.

Current browser capabilities:

- section navigation
- collapsible sections
- component anchors
- prominent finding cards
- semantic recommendation rendering
- semantic table preview rendering
- semantic visualization placeholders
- evidence-link objects
- compact, balanced, and comfortable density handling
- graceful invalid-contract messaging

Intentional non-goals remain:

- no PDF/DOCX/PPTX/HTML export renderer
- no print renderer
- no report editing
- no artifact inspector integration
- no service-specific rendering branches

### Phase 5: Static HTML Fallback

Render the same contract to static HTML with recorded degradation metadata.

### Phase 6: Export Target Evaluation

Add DOCX/PDF/PPTX only after the contract and HTML renderer prove stable.

## Non-Goals

This architecture does not require:

- rebuilding all reports now
- replacing the Collector
- replacing Artifact Studio
- replacing Layout Studio immediately
- generating reports from collector DOCX files
- forcing artifacts to become report components
- implementing every renderer at once
- package extraction before the contract stabilizes
- Studio editing before automatic reports work

## QA Strategy

Minimum QA for the first implementation:

- contract constructor returns valid structure
- validator catches missing required fields
- version fields are present
- components validate by type
- table components validate table contracts
- renderer registry rejects unsupported targets
- static fallback metadata exists for interactive components
- report adapter does not recompute analysis
- report contract can serialize to JSON and restore
- multiple service adapters validate through the same runtime
- current app still sources
- documentation is discoverable in Knowledge Library

## Risks

- Recreating RMarkdown under a new name.
- Mixing presentation profile into semantic contract.
- Letting renderers reinterpret results.
- Treating artifacts as the report model.
- Building Studio editing before automatic reports are reliable.
- Adding a generic workflow engine when a small contract is enough.

## Open Questions

- Should report contracts be persisted in project storage before or after renderer output exists?
- Should `PresentationProfile` live in the future reporting package or the app theme system first?
- Which renderer should become the first export target after Workstation HTML?
- How much of the existing report-plan UI should remain once contract-backed reports exist?
- Should report contracts cite artifact IDs, collector bundle IDs, or both?
